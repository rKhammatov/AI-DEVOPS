Помоги настроить reverse proxy на тестовом VPS-сервере. У меня есть SSH-доступ к серверу, и все команды на самом сервере я буду выполнять сам. Твоя задача — подготовить файлы в папке nginx-devops этого репозитория и дать мне точные команды и пошаговые инструкции; сам на сервере ничего не запускай и не предполагай, что у тебя есть к нему доступ.

## Контекст
- На сервере уже работают три сервиса в Docker: openclaw, ouroboros и mvp (мой основной сервис).
- openclaw — это self-hosted AI-ассистент, наружу смотрит его Gateway (Control UI поверх websocket). У него есть особенности за прокси, см. отдельный раздел ниже.
- Все три сервиса используют websocket. У openclaw и ouroboros ws-соединения идут через их единственный порт (тот же апстрим, что и HTTP). У mvp — отдельный порт под ws. Поддержка Upgrade/Connection обязательна во всех server-блоках.
- ОС — RHEL-подобный дистрибутив (пакеты через dnf/yum, сервисы через systemd, SELinux скорее всего в режиме enforcing).
- Есть второй (боевой) сервер — те же сервисы; конфиги должны легко переноситься (отличаться только поддоменами и портами).
- Нужно поставить перед сервисами nginx как reverse proxy с HTTPS.
- Сетевую изоляцию (файрвол, бинды портов контейнеров) я делаю сам — в инструкции это не расписывай. Весь внешний трафик заходит через внешний прокси.

## RHEL-специфика (важно)
- Раскладка конфигов как принято на RHEL: всё в /etc/nginx/conf.d/*.conf (никаких sites-available/sites-enabled).
- SELinux: по умолчанию он ЗАПРЕЩАЕТ nginx исходящие соединения к апстримам — proxy_pass будет падать с 502/permission denied. В README обязательный шаг: setsebool -P httpd_can_network_connect 1. Также: правильные контексты на файлы сертификата в /etc/nginx/ssl/ (restorecon после копирования) и заметка, что при нестандартных listen-портах понадобится semanage port. Добавь в README мини-раздел "если 502 — проверь SELinux" (как смотреть /var/log/audit/audit.log, ausearch -m avc).

## Архитектура TLS (важно)
- ПЕРЕД моим сервером стоит внешний прокси/балансировщик, на котором живёт настоящий сертификат с поддоменами (*.sblabs.ru). Он принимает трафик из интернета и пересылает его на мой сервер.
- На моём хосте nginx использует САМОПОДПИСАННЫЙ сертификат — он шифрует только участок «фронт → мой сервер». Никакого certbot/Let's Encrypt в проекте быть не должно.
- Дай команду openssl для генерации самоподписанного сертификата: с SAN на *.sblabs.ru (чтобы фронт мог при желании проверять имя), сроком на несколько лет, ключ и серт в /etc/nginx/ssl/ (права на ключ 600, владелец root). Пути пропиши в конфигах с комментарием "# ЗАМЕНИ: ...".
- Маршрутизация по поддоменам sblabs.ru: nginx различает сервисы по server_name, поэтому фронт обязан передавать заголовок Host без изменений — добавь в README пункт проверки этого.
- Какие именно поддомены на тестовом сервере — спроси меня.
- nginx слушает 443 (основной трафик, с http2) и 80. На 80 — только редирект на https, никакого проксирования.
- Реальный IP клиента приходит от фронта в X-Forwarded-For: настрой realip-модуль — real_ip_header X-Forwarded-For + set_real_ip_from <IP фронта> (помети "# ЗАМЕНИ: ...", IP фронта спроси у меня). Доверять X-Forwarded-For только от фронта, не от всего мира.
- Добавь default_server-блок (на 80 и 443), который отвечает 444 на запросы по голому IP или с чужим Host. Для 443-варианта используй тот же самоподписанный сертификат.

## Ключевые решения (придерживайся их)
- nginx ставим ПРЯМО НА ХОСТ (dnf), не в Docker. Сами сервисы остаются в Docker.
- nginx проксирует на сервисы по 127.0.0.1:<порт>.
- Конфиги nginx я правлю сам в /etc/nginx/conf.d. Перезагрузка: `systemctl reload nginx` (её я запускаю сам).

## Что подготовить (файлы в nginx-devops/)
1. Конфиги nginx: по одному файлу на каждый поддомен. В каждом: server-блок на 80 с редиректом на https и основной server-блок на 443 ssl (http2) с proxy_pass на 127.0.0.1:<порт>; корректные заголовки (Host, X-Forwarded-For, X-Forwarded-Proto); обязательная поддержка websocket на всех трёх: proxy_http_version 1.1 + заголовки Upgrade/Connection. Для mvp развести трафик между двумя апстримами: HTTP-порт и websocket-порт — уточни у меня, как отличать ws-трафик (по пути или по заголовку Upgrade).
2. Таймауты: сервисы стримят и держат долгие ws-соединения. На обычных проксирующих локациях proxy_read_timeout / proxy_send_timeout ~300s; на ws-локациях больше (например 3600s), иначе nginx будет рвать соединения, в которых нет трафика. proxy_buffering на стримящих/ws-локациях отключить, иначе будут обрывы и 502.
3. client_max_body_size поставь с запасом (например 50m) — спроси меня, грузятся ли в сервисы файлы и какого размера.
4. Современные TLS-настройки в HTTPS-блоках (TLSv1.2/1.3, нормальные ciphers) — без экзотики, с комментарием.
5. README с пошаговой инструкцией: что и в каком порядке мне выполнить на сервере (установка nginx, SELinux-шаги, генерация самоподписанного сертификата, какой файл из nginx-devops/ куда копируется в /etc/nginx/conf.d, nginx -t, reload), с готовыми командами, которые я просто скопирую. Плюс короткие разделы: как перевыпустить самоподписанный сертификат, когда истечёт (команда + reload); чек-лист настроек фронта (передавать Host без изменений, проксировать websocket, слать X-Forwarded-For, принимать самоподписанный сертификат апстрима или отключить его проверку); раздел "перенос на боевой сервер" — что именно поменять.

## Особенности openclaw (проверено по официальной документации docs.openclaw.ai)
- Gateway по умолчанию слушает loopback на порту 18789, WebSocket и HTTP идут через один порт — nginx проксирует всё одним апстримом 127.0.0.1:<порт openclaw> с поддержкой Upgrade.
- В конфиге openclaw (~/.openclaw/openclaw.json) указать nginx как доверенный прокси: gateway.trustedProxies: ["127.0.0.1"]. Без этого Gateway не доверяет прокси-заголовкам и ломаются проверки локального клиента.
- В server-блоке openclaw перезаписывать заголовки, как требует их документация: proxy_set_header X-Forwarded-For $remote_addr; и X-Real-IP $remote_addr; (НЕ использовать $proxy_add_x_forwarded_for — Gateway не должен получать дописанные чужие XFF). С учётом realip-модуля (set_real_ip_from на IP фронта) $remote_addr будет настоящим IP клиента.
- Публичный URL Control UI (https://<поддомен-openclaw>.sblabs.ru) добавить в gateway.controlUi.allowedOrigins — для не-loopback доступа к Control UI это обязательная настройка.
- Аутентификация — родная для openclaw: gateway.auth.mode: "token" (или password). НЕ использовать режим trusted-proxy: для loopback-прокси он намеренно fail-closed, и доп. авторизацию на прокси мы не делаем.
- Не форвардить через nginx identity-заголовки (например tailscale-user-login).
- Control UI требует secure context — у нас он обеспечен HTTPS-сертификатом на фронте.
- Настройки openclaw я применю сам — дай готовый фрагмент для openclaw.json и команды openclaw config set.
